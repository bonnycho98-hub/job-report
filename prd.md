## 1. 개요

채용공고 사이트들을 **매주 자동으로 크롤링**하여, 사용자의 프로필/조건에 맞는 공고만 필터링해 **웹페이지로 보여주는 서비스**.

- **사용자**: 치호 + 여자친구 (2인 사용, 프로필 분리)
- **핵심 가치**: 매주 수동으로 여러 사이트를 돌아다니는 시간을 0으로 줄인다
- **채용공고 사이트**: 사용자가 직접 등록/관리

---

## 2. 문제 정의

| 문제 | 현재 상태 |
| --- | --- |
| 여러 채용 사이트를 매번 수동 확인 | 시간 소모 큼, 누락 발생 |
| 관심 없는 공고까지 다 봐야 함 | 피로도 증가 |
| 새 공고를 놓치면 기회 상실 | 주기적 확인이 강제됨 |

---

## 3. 목표

1. 등록된 사이트를 **수동 버튼 클릭으로 크롤링** (PC에서 실행)
2. 사용자 조건에 맞는 공고만 **자동 필터링**
3. 필터링된 공고를 **웹페이지에서 한눈에 확인**

---

## 4. 사용자 시나리오

### 4-1. 채용 사이트 등록

1. 사이트별 **전용 파서 모듈**을 코드로 작성 (CSS 선택자 / XPath 등)
2. 새 사이트 추가 시 파서 모듈 1개 추가 개발 필요
3. 파서 모듈 구조를 통일하여 새 사이트 추가 시 최소한의 코드만 작성

### 4-2. 매칭 조건 설정

1. 사용자가 관심 키워드, 직무, 지역, 경력 등 필터 조건을 설정
2. 조건은 언제든 수정 가능

### 4-3. 크롤링 실행 & 결과 확인

1. 사용자가 웹페이지에서 **"크롤링 시작" 버튼 클릭**
2. 크롤링 → 파싱 → 매칭 → 결과 저장
3. 완료 후 매칭 공고 목록이 화면에 표시
4. 과거 실행 결과도 아카이브로 열람 가능

## 5. 핵심 기능

### 5-1. 사이트 관리

- 크롤링 대상 사이트 URL 등록/삭제
- 사이트별 상태 표시 (정상 / 크롤링 실패 / 구조 변경 감지)
- 사이트별 전용 파서 모듈 (코드 레벨로 관리)

### 5-2. 크롤러

- **수동 트리거**: 웹페이지 내 버튼 클릭으로 실행 (로컬 PC에서 동작)
- 사이트별 전용 파서 모듈이 HTML 파싱 → 공고 데이터 추출
    - 공고 제목, 회사명, 직무, 마감일, URL, 게시일
- 중복 공고 제거 (URL 기준 dedup)
- **실패 시 상세 에러 리포팅**:
    - 사이트별 성공/실패 상태 구분 표시
    - 실패 사이트: 에러 유형 (HTTP 에러 / 선택자 불일치 / 타임아웃 / 파싱 실패)
    - 에러 발생 위치 (어떤 선택자에서 실패했는지)
    - 마지막 성공 시점 vs 현재 실패 시점 비교
    - UI에 사이트별 에러 로그 확인 가능

### 5-3. 매칭 엔진

공고를 **2개 프로필**로 분류하여 각각 독립적으로 매칭한다.

**프로필 A: 마케팅 (여자친구 용)**

- 키워드 (제목/본문에 1개 이상 포함 시 매칭):
    - `마케팅`, `Marketing`, `프로모션`, `promotion`, `Onsite`

**프로필 B: 서비스 운영 기획 (치호 용)**

2개 서브 그룹으로 분류:

*그룹 B-1: 서비스 운영 직무*

- `서비스운영`, `서비스 운영`, `Service Operations`, `Operations Manager`, `운영기획`, `운영 기획`, `운영관리`, `정책 운영`, `정책 설계`, `프로세스 개선`, `프로세스 설계`, `BizOps`, `Business Operations`, `업무 효율화`, `운영 자동화`, `VOC`,`Product Operations`

*그룹 B-2: PM 공고*

- `PM`, `Product Manager`, `프로덕트 매니저`, `Growth PM`, `Growth Operations`, `Program`

*그룹 B-3: 기획 공고*

- `기획자`, `서비스기획`, `서비스 기획`

*그룹 B-4: CS / CX 공고*

- `CS`, `CX`, `Customer Success`, `Customer Experience`, `고객경험`, `고객 경험`, `고객성공`, `고객 성공`, `고객지원`, `고객 지원`

**매칭 규칙**

- 각 프로필별로 독립 매칭 → 결과 페이지에서 탭으로 분리 표시
- 프로필 B는 서브 그룹(B-1, B-2, B-3, B-4) 표시로 어떤 유형의 공고인지 구분
- 매칭 점수 산출: 키워드 충족 개수 기반
- 향후 확장: LLM 기반 시맨틱 매칭

### 5-4. 웹페이지 (프론트엔드)

**메인 대시보드**

- **탭 구분**: 마케팅 / 서비스 운영 기획 (각 프로필별 매칭 결과 분리 표시)
- 서비스 운영 기획 탭 내에서 B-1(운영 직무) / B-2(PM) / B-3(기획) / B-4(CS·CX) 서브 그룹 표시
- 각 공고 카드: 회사명, 직무, 마감일, 매칭 점수, 원문 링크
- 주차 선택으로 과거 결과 탐색

**사이트 관리 페이지**

- 등록된 사이트 목록
- 사이트 추가/삭제
- 크롤링 상태 확인

**설정 페이지**

- 매칭 키워드/조건 관리

### 5-5. HTML 파일 내보내기

- 매칭 결과를 **정적 HTML 파일로 출력**하는 기능
- 웹페이지 내 "파일 내보내기" 버튼 클릭 → `output/YYYY-MM-DD.html` 생성
- HTML 파일 포함 내용:
    - 프로필별(마케팅 / 서비스 운영 기획) 매칭 공고 리스트
    - 회사명, 직무, 마감일, 매칭 점수, 원문 링크
    - 크롤링 실행 일시 / 사이트별 성공·실패 요약
- 외부 의존성 없이 **브라우저로 바로 열 수 있는** 단일 파일
- 용도: 오프라인 백업, 공유, 아카이브

## 6. 기술 스택 (제안)

| 구분 | 기술 |
| --- | --- |
| 크롤러 | Python (requests + BeautifulSoup / Playwright) |
| 백엔드 | FastAPI 또는 Flask |
| DB | SQLite (초기) → PostgreSQL (확장 시) |
| 프론트엔드 | Next.js 또는 정적 HTML + Tailwind |
| 배포 | 로컬 실행 (추후 배포 가능) |

## 7. 데이터 모델

```jsx
# 크롤링 사이트
Site {
  id, url, name, status, selector_config, created_at, updated_at
}

# 크롤링된 공고
JobPosting {
  id, site_id, title, company, position, location, 
  experience_level, employment_type, deadline, 
  source_url, posted_at, crawled_at
}

# 매칭 프로필
Profile {
  id, name, keywords, sub_groups
}

# 매칭 결과
MatchResult {
  id, job_posting_id, profile_id, sub_group,
  match_score, matched_keywords, 
  week_number, year, created_at
}
```
